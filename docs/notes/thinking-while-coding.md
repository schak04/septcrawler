# Thinking while Coding

To avoid getting stuck while implementing, I've settled on the following approach:

1. I'll first think through the problem using temporary comments or pseudocode (or by writing things down).
2. If I stop working before solving it, I'll leave WIP comments documenting the current problem, my assumptions, thought process, and next steps. I may commit these if needed and remove them once the work is complete.
3. When I need to visualise a problem, I'll sketch it and do whatever else is needed using pen and paper (or my tablet), not a mouse-based whiteboard application, because I've realised those slow me down. I also get fixated on making the sketches look good, which creates a lot of friction in my thinking.
4. I'll keep sketches lightweight and temporary, and it is easy to do so while using pen and paper. Their purpose is to help me think, not to become documentation.
5. I always create proper diagrams (e.g. in draw.io) when they are intended as design documentation, alongside textual documentation, and I will keep doing so.
